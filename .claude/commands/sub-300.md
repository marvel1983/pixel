# /sub-300 — Audit fajlova koji se približavaju 300-line cap-u

Projekt ima striktno pravilo: **svi source fajlovi ispod 300 linija**. Ovaj command pronalazi fajlove koji su blizu ili preko limita i predlaže kako ih podijeliti.

## Koraci

1. **Skeniraj sve TS/TSX fajlove u artifacts/ i lib/:**
   ```bash
   find artifacts lib -type f \( -name "*.ts" -o -name "*.tsx" \) \
     ! -path "*/node_modules/*" \
     ! -path "*/dist/*" \
     ! -path "*/lib/api-client-react/*" \
     ! -path "*/lib/api-zod/*" \
     -exec wc -l {} + | sort -rn | awk '$1 > 250 && $1 != "total"' | head -30
   ```

2. **Klasificiraj svaki rezultat:**
   - **300+** → MORA se podijeliti prije sljedećeg commit-a
   - **250–299** → upozorenje, razmisliti o split-u
   - **<250** → OK

3. **Za svaki problematični fajl predloži split strategy:**

   | Pattern u fajlu | Split strategija |
   |---|---|
   | Više `export function`-a koje rade različite stvari | Izdvoji u `foo-helpers.ts`, `foo-utils.ts` |
   | Velika React komponenta sa više sub-sectiona | Izdvoji sub-komponente u `foo/`, koristiti index.tsx |
   | Zustand store sa mnogo actions | Izdvoji actions u `foo-actions.ts`, store import |
   | Route fajl sa mnogo endpointa | Split po resource: `admin-foo.ts`, `admin-foo-bulk.ts` |
   | Schema fajl sa više tabela | Po tabeli — jedan fajl = jedna tabela |
   | Service sa više concern-a | Split po concern: `pricing-flash.ts`, `pricing-bundle.ts` |

4. **Prikazi rezultat u tabeli i pitaj korisnika šta da radi.**

## Izuzeci (ne broje se)

- `lib/api-client-react/**` — generisan kod
- `lib/api-zod/**` — generisan kod
- `**/*.test.ts`, `**/*.spec.ts` — testovi mogu biti duži
- JSON fixture fajlovi
- Type-only `.d.ts` fajlovi koji su deklaracije
